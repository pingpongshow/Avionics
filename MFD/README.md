# MFD Module Specification

## Purpose

The `MFD` module is the primary display and integration node for the system.

It is responsible for:

- displaying flight, navigation, engine, traffic, and system data
- hosting the moving map and ownship position
- adapting layouts based on connected modules
- providing user controls, menus, setup pages, and alerts
- supporting a `Demo Mode` for bench testing and development

It is not responsible for directly measuring primary air-data, attitude, or engine probes. Those come from dedicated external modules.

## Role In The System

The `MFD` should be designed as one display node in a modular distributed system.

### MFD local responsibilities

- local GPS/GNSS input
- local map database storage
- local display, controls, and UI rendering
- system configuration and module status presentation
- command and configuration forwarding to other modules

### MFD external dependencies

- `ADC/AHRS` for airspeed, altitude, VSI, pitch, roll, slip/skid, OAT, and optional AOA
- `EIS` for engine and fuel data
- `ADSB` for traffic and optional future weather
- `system bus` for discovery, health, and control

## MFD Functional Requirements

### Minimum required functions

- boot into a usable display without requiring all optional modules
- show moving map and ownship when only `GPS` is available
- show full PFD when `ADC/AHRS` is connected
- add engine strip/pages when `EIS` is connected
- add traffic overlays/pages when `ADSB` is connected
- provide clear fault indication when required data is missing or stale
- support day/night brightness control
- support `Demo Mode`

### Primary display modes

- `PFD Focus`
- `PFD + Map Split`
- `PFD + Map + Engine`
- `Full Map`
- `Engine Page`
- `Traffic Page`
- `Reversionary`
- `Standby / Reduced Layout`

### UI behavior requirements

- layouts must adapt automatically to module presence
- user must be able to save layout presets
- module failure must collapse gracefully to a simpler layout
- critical flight data must remain readable with engine and map panes enabled
- the same hardware should be configurable as `Primary MFD`, `Primary PFD`, `Map Repeater`, `Engine Display`, or `Standby Display`

## Information The MFD Needs

The MFD is only as good as the data contracts used by the other modules. Before implementation, the following information needs to be defined for every module.

### Required for each external data source

- message name / ID
- units
- numeric range
- update rate
- timestamp source
- validity flag
- stale timeout
- failure mode / fault code
- source priority if redundant modules exist

### ADC/AHRS data required

- pitch
- roll
- yaw rate / turn rate
- slip/skid ball or lateral acceleration estimate
- indicated airspeed
- pressure altitude
- baro-corrected altitude
- vertical speed
- baro setting state
- outside air temperature
- true airspeed if available
- density altitude if available
- AOA if installed
- sensor health and alignment/calibration status

### GPS/GNSS data required

- latitude / longitude
- ground speed
- ground track
- UTC time
- fix quality
- satellites used
- estimated position accuracy
- altitude source and accuracy

### EIS data required

- RPM
- manifold pressure
- oil pressure
- oil temperature
- fuel pressure if installed
- fuel flow
- fuel used / remaining if provided externally
- CHT per cylinder
- EGT per cylinder
- volts / amps
- warning / caution states

### ADSB data required

- traffic target ID
- relative / absolute position
- altitude
- track / ground speed if available
- target age
- threat state if generated by the ADSB module
- optional weather product type if weather is added later

### System data required

- module heartbeat
- module type
- module serial / version
- firmware revision
- capability list
- temperature / supply / internal fault status

## Recommended MFD Hardware

## v1 baseline

- `Compute`
  `Raspberry Pi CM5`
- `RAM`
  `8 GB` minimum, `16 GB` preferred if terrain, traffic, caching, or larger map features grow
- `Display`
  `10.1"` sunlight-readable IPS display, target `1280x800`, target `1000 nits`
- `Storage`
  eMMC or NVMe preferred, no dependency on microSD for installed use
- `GPS/GNSS`
  local receiver on the MFD
- `Communications`
  dual `CAN FD`, USB service, optional Ethernet, optional serial debug
- `Controls`
  two rotary encoders with push, 6-8 soft keys, menu/back keys, brightness access

## Hardware recommendations

### Compute

Recommended first platform:

- `Raspberry Pi CM5`

Reason:

- enough graphics and CPU headroom for split-screen map and instrument rendering
- good Linux support
- easy prototype path
- reusable for primary and secondary display nodes

### Display

Recommended first target:

- `10.1"` display
- `1280x800`
- `1000 cd/m2`
- wide viewing angle IPS panel

Reason:

- good fit for PFD + map split layouts
- close to the display class used by many successful experimental EFIS products
- easier to preserve readable tapes and map area simultaneously than on smaller screens

### GNSS

Recommended first target:

- local `u-blox NEO-F10N` class receiver

Reason:

- keeps map and ownship smooth and independent of other modules
- simplifies initial bring-up
- provides local time source for logs and playback

### Controls

Recommended front-panel control set:

- left encoder with push
- right encoder with push
- 6 to 8 soft keys
- dedicated `Menu`
- dedicated `Back`
- optional `NRST / nearest` shortcut
- optional brightness shortcut

Touch should be treated as optional, not required.

### Audio

Not required for the first MFD build unless alert tones or intercom control are explicitly added.

### Power

The MFD should not be powered like a hobby desktop display.

The design needs:

- aircraft input power stage
- transient protection
- reverse polarity protection
- filtered DC rails
- supervised power-fail input
- orderly shutdown or ride-through behavior

### Thermal

The CM5 enclosure and display bezel need an intentional thermal design.

At minimum:

- heat spreading path from SoM
- venting or passive metal chassis conduction
- temperature monitoring and thermal warning path

## Recommended MFD Software Stack

## v1 software recommendation

- `OS`: Linux
- `UI framework`: `Qt 6 / QML`
- `backend language`: `C++`
- `graphics`: Qt scenegraph with GPU acceleration
- `IPC / internal messaging`: typed in-process pub/sub or message bus
- `module bus interface`: `CAN FD`

Reason:

- Qt/QML is well suited to embedded HMI work
- adaptive layouts and instrument composition are easier than with lower-level graphics alone
- C++ gives direct access to Linux device, serial, CAN, and graphics tooling

## Core software components

### Boot/runtime

- watchdog support
- fast startup path
- read-only or resilient root filesystem strategy
- persistent config partition
- log capture and crash diagnostics

### UI engine

- layout manager
- widget library for tapes, HSI, annunciators, gauges, strip charts, and engine bars
- theme/day-night palette system
- alert presentation layer
- font and scaling rules for multiple screen sizes

### Navigation/map

- ownship tracking
- local chart/map database
- pan/zoom logic
- declutter levels
- waypoint / route rendering
- terrain shading later if desired

### Data integration

- module discovery
- message decode / validation
- stale-data handling
- redundancy handling later
- data recording and playback

### Configuration

- display role selection
- brightness setup
- demo mode setup
- module status page
- software version page
- installation/configuration page

### Demo Mode

The first version should include simulation hooks from day one.

Required demo capabilities:

- synthetic GPS only
- synthetic ADC/AHRS
- synthetic EIS
- synthetic ADSB traffic
- scenario presets
- on-screen `DEMO` marking

## Map Data Strategy

The MFD owns map presentation, so map data strategy must be chosen early.

### Current source policy

- `FAA` official chart packages should be the primary U.S. moving-map source
- `OpenFlightMaps` should be the preferred regional chart source where it has coverage
- `openAIP` should be treated as a supplemental overlay data source, not the primary chart
- `VFRMap` is a good reference site, but should not be used as the project backend data source

### First-build recommendation

Start with packaged offline raster aviation charts rather than a web map or a full custom vector avionics renderer.

Preferred first path:

- `FAA` `GeoTIFF` VFR charts imported into a local chart package format for U.S. regions
- `OpenFlightMaps` `MBTiles` or `GeoTIFF` packages for supported non-U.S. regions
- ownship overlay
- route and waypoint overlay
- `ADSB` traffic overlay
- optional `openAIP` supplemental overlays where license and use case allow it

This keeps the first version focused on avionics integration rather than becoming a GIS project or depending on live internet map services.

## MFD Interfaces

### Inputs

- GNSS serial/UART or USB
- CAN FD bus A
- CAN FD bus B preferred
- front-panel GPIO / encoder inputs
- optional ambient light or dimming input
- optional USB service port

### Outputs

- displayed information
- module configuration commands
- display-role broadcast
- health/status messages
- alert acknowledgements
- logs for support and playback

## Fault Handling Requirements

The MFD must treat missing data as a first-class condition.

### Required fault behaviors

- stale data timeout per signal family
- visual invalid indication for failed sources
- degrade gracefully from full layout to partial layout
- never silently reuse old data as live data
- detect loss of GPS separately from loss of ADC/AHRS
- detect module absence separately from module internal fault

## Recommended Development Phases

### Phase 0

- bring up display head only
- implement `Demo Mode`
- build layout engine

### Phase 1

- add local GPS and moving map
- validate storage, boot, and UI controls

### Phase 2

- integrate `ADC/AHRS`
- enable PFD layouts and failure handling

### Phase 3

- integrate `EIS`
- enable engine strips and engine pages

### Phase 4

- integrate `ADSB`
- enable traffic overlays and traffic page

### Phase 5

- add secondary display roles and reversionary behavior

## Open Design Decisions

- exact display panel and bezel dimensions
- exact GNSS module and antenna arrangement
- final CAN FD message schema
- whether one or two CAN buses are required in v1
- whether the first map engine is raster, vector, or hybrid
- whether touch is omitted entirely in v1
- whether ADSB is a USB-attached prototype first or native bus module first
